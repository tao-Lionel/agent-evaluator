"考卷"长什么样？

每道题是一个 JSON 对象，描述了初始状态、用户消息和期望结果：

JSON 
{
  "id": "order-cancel-001",
  "description": "用户想取消一个待处理的订单。",
  "initial_message": "你好，我想取消订单 ORD-1001。",
  "initial_state": {
    "orders": [
      {"id": "ORD-1001", "status": "pending"}
    ]
  },
  "required_info": ["ORD-1001", "取消"]
}
人话翻译

"id" — 这道题的编号，方便后面对照成绩单。

"description" — 题目描述：用户要干什么。

"initial_message" — 用户发出的第一句话。

"initial_state" — 考试开始前数据库里已有的数据。

"required_info" — AI 的回复里必须包含这些关键词才算合格。

💡
为什么自动化评估很重要？

AI 模型每次更新都可能"退步"——之前能做对的事情突然做错了。这叫回归。自动化评估就像定期模拟考试，每次更新都跑一遍，立刻发现退步的地方。

每个角色的"合同"

在代码中，每个角色都有一个抽象基类，定义了它必须能做什么：

🔌

AgentAdapter

reset() 重置 & act(messages) 生成响应。不管是 OpenAI、HTTP 还是 WebSocket，都必须实现这两个方法。

🏗

Environment

reset(task) 初始化 & step(tool_call) 执行工具。提供考场环境和可操作的工具。

📊

Evaluator

evaluate(task, trajectory, env) 返回 0.0~1.0 的分数。每个评估器专注一个维度。

👤

UserSimulator

respond(task, trajectory) 返回用户回复或 None（表示满意）。驱动多轮对话。

💡
关注点分离

每个角色只做自己的事——Adapter 不关心怎么打分，Evaluator 不关心怎么调 API。这种设计叫"关注点分离"，是软件工程最重要的思想之一。

插件系统——装饰器魔法

一行代码注册，全局自动可用

便利贴和储物柜

想象你进入一个储物柜房间。每个柜子上有标签："适配器"、"环境"、"评估器"、"用户模拟器"。你只需要把新物品贴上便利贴写上名字，放进对应的柜子，任何人来找的时候报名字就能取到。

在代码中，这个储物柜就是 Registry，便利贴就是 装饰器。

Registry 的核心代码

CODE 
class Registry:
    def __init__(self):
        self._adapters: dict[str, type] = {}
        self._environments: dict = {}
        self._evaluators: dict = {}
        self._users: dict = {}

    def adapter(self, name: str):
        def wrap(cls):
            self._adapters[name] = cls
            return cls
        return wrap
人话翻译

__init__ — 准备四个空储物柜：适配器柜、环境柜、评估器柜、用户柜。

_adapters: dict — 每个柜子就是一个字典：名字 → 组件类。

adapter(name) — 返回一个"便利贴生成器"。

wrap(cls) — 真正的"贴标签"动作：把类存进柜子，名字是你指定的。

return cls — 物品（类）本身不变，只是多了一张标签。

一行装饰器 = 自动注册

有了 Registry，注册一个新适配器只需要一行代码：

CODE 
@registry.adapter("openai_fc")
class OpenAIFCAdapter(AgentAdapter):
    # 具体实现...
    pass
人话翻译

@registry.adapter("openai_fc") — 在适配器柜子里放一张便利贴，写着 "openai_fc"。

class OpenAIFCAdapter — 这就是放进去的物品（具体的适配器类）。

以后任何人只需说 registry.get_adapter("openai_fc")，就能取到这个类。

评估器打分

核心循环代码

CODE 
for step in range(task.max_steps):
    try:
        agent_msg = self.agent.act(trajectory)
    except Exception as e:
        termination = TerminationReason.AGENT_ERROR
        break

    trajectory.append(agent_msg)

    if not agent_msg.tool_calls:
        if self.user:
            user_msg = self.user.respond(task, trajectory)
            if user_msg is None:
                termination = TerminationReason.USER_STOP
                break
        continue

    for tc in agent_msg.tool_calls:
        result = self.env.step(tc)
        if result.done:
            task_done = True
人话翻译

for step in range(max_steps) — 最多让 Agent 走 N 步，防止无限循环。

agent.act(trajectory) — 把完整对话历史发给 Agent，让它决定下一步。

except → AGENT_ERROR — Agent 崩溃了？记录错误，优雅退出。

if not tool_calls — Agent 只是说话，没调用工具。

user.respond() is None — 模拟用户满意了，对话结束。

env.step(tc) — 把工具调用交给环境执行（如修改数据库）。

result.done — Agent 调用了 done 工具，任务完成。

乘积评分

💡
为什么是乘积而不是平均？

overall = s1 × s2 × s3。任何一个维度得 0 分，总分就是 0。就像飞机起飞前的安全检查——发动机正常但起落架坏了，也绝对不能起飞。这种"一票否决"的设计借鉴了 τ²-Bench 的研究。

打分的艺术

六位考官，各有绝活

考官团队

🗃

state_match

比对数据库最终状态。先用 MD5 精确匹配，失败后尝试子集匹配。

🔧

action_match

检查工具调用是否正确。支持部分参数匹配。

💬

info_delivery

回复中是否包含所有必要信息。模糊子串匹配，容忍格式差异。

🤖

llm_judge

用 LLM 对回复质量打 1-5 分。支持 INSUFFICIENT_INFO 逃生门。

✅

nl_assertion

自然语言断言，LLM 逐条判 PASS/FAIL。

🛡

safety

安全性检查：有害内容、越权访问等。

state_match：两阶段验证

人话翻译

先用预期操作重建"标准答案"数据库。

对两个数据库做 MD5 指纹比对。指纹一样 = 内容一致 = 满分。

精确匹配失败？试试宽松模式：只要关键字段对了就给部分分。

连接外部世界

三种适配器，适配任何 AI Agent

联合国翻译官

适配器是框架与外部 AI 之间的翻译官。框架说统一语言，每种 AI 有自己的方言。

⚡
openai_fc

适配 OpenAI 兼容的函数调用 API（智谱、Azure 等）

🌐
http_bot

适配任意 HTTP API，通过 request_template 配置驱动

🔌
ws_bot

适配 WebSocket 流式 Agent，支持超时控制

消息格式转换

openai_fc 适配器的核心工作：把框架内部的 Message 对象翻译成 OpenAI API 需要的 JSON 格式。

Role.SYSTEM→ {"role": "system"} — 传递系统提示词
Role.USER→ {"role": "user"} — 用户的消息
Role.AGENT→ {"role": "assistant"} — Agent 的历史回复
Role.ENV→ {"role": "tool"} — 工具执行结果
