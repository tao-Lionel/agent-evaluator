为什么要评估 Agent？

Agent = LLM + 工具调用 + 环境交互 + 多轮决策

一个好的 LLM 不一定是一个好的 Agent。模型在基准测试上得分很高，但接入工具后可能频繁产生非法调用、忘记上下文、或无法完成多步任务。

维度四 & 五

回复质量 & 自然语言断言

LLM Judge

用 LLM 对回复打分 (1-5)，归一化为 0~1

  • 准确性 — 信息是否正确？
  • 完整性 — 是否遗漏关键内容？
  • 清晰度 — 表达是否易懂？
  • 一致性 — 是否自相矛盾？
特别适用于黑盒 Agent — 回复质量可能是唯一能评估的维度

NL Assertion

用自然语言描述评判条件，LLM 逐条判 PASS/FAIL

// 客服 Agent 断言 PASS 取消前确认了用户意图 PASS 未泄露其他客户信息 FAIL 未提供替代方案 score = 2/3 = 0.67

维度六

一致性与稳定性 Pass^k

对每个任务运行 n 次，衡量 Agent 的稳定性而非单次偶然成功

Pass^1 高 + Pass^k 低

有能力但不稳定 — 最危险，给人虚假信心

Pass^1 低

能力不足 — 问题明确，至少不会误导

Pass^1 高 + Pass^k 高

既有能力又稳定 — 生产环境可信赖

你不会信任一个"有时能正确写代码，有时写出 bug"的编码 Agent。
08 / 14
AgentBoard 启发

过程指标：不只看结果

进度率 Progress Rate

即使任务未完成，也追踪完成了多少子步骤

Agent A: 搜索→排序→选品→加购 PR=1.0 Agent B: 搜索→排序→选品→放弃 PR=0.75 Agent C: 搜索→(陷入循环) PR=0.25 Agent D: (什么都没做) PR=0.0

接地准确度 Grounding

合法动作数 / 总动作数

衡量 Agent 是否"知道自己能做什么" — 发出非法指令说明对工具理解有误

关键价值：区分"完全失败"和"做了一半"的 Agent，为改进指明方向

为什么用乘积而非加权平均？

加权平均

0.4×state + 0.3×action + 0.3×info state=0, action=1.0, info=1.0 overall = 0.6 什么都没做对，还能 60 分？

乘积组合 ✓

state × action × info state=0, action=1.0, info=1.0 overall = 0.0 任一维度为零，总分归零。
场景stateactioninfooverall
代码写对但没解释1.01.00.00.0
解释清楚但没执行0.00.01.00.0
查询正确，解释清楚1.01.01.01.0
操作正确但漏了步骤1.00.51.00.5

通用评估架构：Agent-Environment 循环

┌──────────┐ 初始化 → │ Agent │ ←──────┐ │ (被评估者) │ │ └─────┬────┘ │ │ 观察/反馈 动作/回复 │ │ │ ▼ │ ┌──────────────┐ │ │ Environment │────┘ │ (任务环境) │ └──────┬───────┘ 任务完成/超时 │ ▼ ┌──────────────┐ │ Evaluators │ │ (多维度评分) │ └──────────────┘

这个循环是通用的 — 区别只在于三个维度的具体实现：

维度变化点
AgentAPI 调用？HTTP 接口？本地进程？
Environment数据库？文件系统？浏览器？
Evaluators状态匹配？测试通过率？回复质量？
核心是可插拔设计：新增 Agent 类型或评估维度只需添加实现，不修改框架

评估方法分类

确定性 vs LLM-as-Judge

确定性评估

  • state_match DB 哈希比对 + 子集比较
  • action_match 工具名 + 参数匹配
  • info_delivery 子串搜索

完全可复现 | 零额外 API 调用

LLM-as-Judge

  • llm_judge LLM 打分 (1-5)
  • nl_assertion LLM 逐条判断 PASS/FAIL

能评估开放式问题 | 需额外 LLM 调用

能用确定性评估的优先用确定性评估（更便宜、更可复现）；无法用规则衡量的维度才用 LLM-as-Judge。

Part 05

实践建议

维度选择决策树

有状态变更？ 是 → state_match + action_match 需要输出信息？ 是 → info_delivery 需要评估主观质量？ 是 → llm_judge 有难以规则化的约束？ 是 → nl_assertion 需要生产可靠性？ 是 → 多次试验 + Pass^k

常见陷阱

  • × 只看成功率 → 多维度评估
  • × 评估不可复现 → temp=0 + Pass^k
  • × 场景太简单 → 分难度 + 边界用例
  • × 忽略过程只看结果 → action_match
  • × 把评估当一次性测试 → 持续评估

总结

Agent 评估的核心公式

Agent 评估 = 多维度 × 多场景 × 多试验

多维度

不只看结果，还看过程、沟通、合规

多场景

覆盖不同难度、不同类型的任务

多试验

衡量稳定性，而非单次偶然成功

乘积组合，而非加权平均。任何一个维度的完全失败都应该导致总分归零。
