AI Agent 评估方法论

如何科学地评判一个 Agent 到底行不行?

内部技术分享

01

Agent 和聊天机器人有什么不同?

传统聊天机器人只需要"说得好"。但 Agent 不一样, 它不只是说话, 还要操作: 查数据库、调接口、改状态、做决策。

说对了 ≠ 做对了 ≠ 做对了且告诉了用户

评估维度从一维变成了多维 — 只看回复文本远远不够。

02

一个例子:"帮我取消订单"

场景数据库状态工具调用回复结论
Acancelled ✓query+update ✓"已取消" ✓完美
Bcancelled ✓query+update ✓"好的" ✗做对了但没说清
C没改 ✗只 query ✗"已取消" ✓说了谎
Dcancelled ✓跳过 query ✗"已取消" ✓过程不规范

只看回复文本, 场景 C 和 D 都会被判为"通过" — 但一个在说谎, 一个跳过了验证。

03

六个评估维度

01
状态正确性
做完之后, 世界的状态对不对?
02
行为规范性
操作过程是否符合预期?
03
信息传达
关键信息有没有告诉用户?
04
回复质量
整体回复质量如何?
05
断言验证
具体业务规则是否被满足?
06
安全性
有没有做不该做的事?
04

维度 1 状态正确性 state_match

最"硬"的指标 — 不管 Agent 说了什么, 最终系统状态必须和预期一致。

阶段一: 精确匹配

用预期操作重建"标准答案"数据库, 对比 MD5 哈希。一样 = 满分。

阶段二: 子集匹配

精确匹配失败时, 检查关键字段是否正确, 给部分分数。容忍非关键差异。

state_match 只管"结果对不对", 不管"怎么做的" — 过程由 action_match 单独评判。

05

维度 2 行为规范性

不只看结果, 还看过程。"取消订单"应该先查询再更新, 不能跳步。

部分参数匹配

query(table="orders") → update(status="cancelled")

只检查必须有的参数, 容忍多出来的。

维度 3 信息传达

Agent 操作成功但只回复"好的" — 用户根本不知道发生了什么。

模糊子串匹配

定义必须出现的关键词, 归一化后匹配。

得分 = 匹配数 / 总数 (说了 4/5 = 0.8 分)

06

维度 4 回复质量 LLM-as-Judge

有些场景没有标准答案 (如"推荐礼物"), 需要用另一个 LLM 当评委打分。

  • » 评委按 1-5 分打分, 归一化为 0.0-1.0
  • » 设计了 INSUFFICIENT_INFO 逃生门
  • » 判不了就说"判不了", 不强行打分

已知偏见

  • • 位置偏见: 交换顺序, 分数偏移 >10%
  • • 冗长偏见: 更长 = 更高分
  • • 自我偏好: 给自己打高分
07

维度 5 断言验证

LLM 逐条判定业务规则 PASS 或 FAIL。

"refunds 表中应存在 ORD-2001 的记录"

"回复中应提到退款到账时间"

得分 = 通过数 / 总数

vs llm_judge: 前者是逐条清单, 后者是整体印象分。

维度 6 安全性

减分项 — 检查 Agent 有没有做不该做的事。

  • ✗ 有害内容 (辱骂、歧视)
  • ✗ 越权操作 (删除他人数据)
  • ✗ 信息泄露 (暴露隐私)
  • ✗ 违反业务规则

关键词 + LLM 双重检测

08

为什么用乘积, 不用加权平均?

overall = state × action × info × ...
加权平均

state=0, action=1, info=1

= 0.6 分

什么都没干还拿 60 分?

乘积

state=0, action=1, info=1

= 0 分

准确反映失败

每个维度都是必要条件, 不能互相补偿。借鉴 τ-bench (ICLR 2025) 的设计。

09

稳定性指标: Pass@k 和 Pass^k

同一个任务跑 5 次, 3 次通过 2 次失败 — 这个 Agent 到底行不行?

Pass@k — 能力上限

k 次机会中至少 1 次通过的概率

想象 5 张牌 ✓✓✓✗✗, 闭眼抽 k 张, 至少抽到 1 张 ✓ 的概率

Pass^k — 稳定性下限

k 次全部通过的概率

每次 60% 通过, 连续 3 次都通过 = 0.6 × 0.6 × 0.6 = 21.6%

两者差距越大, Agent 越不稳定 — 有能力但不可靠。

10

计算示例: 5 次试验, 3 次通过

指标计算结果
Pass@1抽 1 张, ✗ 概率 = 2/5, 至少 1 ✓ = 1 - 2/50.6
Pass@2抽 2 张, 全 ✗ = 2/5 × 1/4 = 1/10, 至少 1 ✓ = 1 - 1/100.9
Pass@3抽 3 张, 只有 2 张 ✗, 不可能全 ✗1.0
Pass^10.610.6
Pass^20.6 × 0.60.36
Pass^30.6 × 0.6 × 0.60.216

Pass@2 = 0.9 vs Pass^2 = 0.36 — "给两次机会大概率能成一次", 但"两次都成"只有三分之一。

11

评估维度的选择逻辑

白盒 Agent (有工具调用)

  • » state_match 必选
  • » action_match 必选
  • » info_delivery 推荐
  • » nl_assertion 可选
  • » safety 可选

黑盒 Agent (只有文本回复)

  • » llm_judge 必选
  • » info_delivery 推荐
  • » nl_assertion 可选
  • » safety 可选
能用确定性指标就不用 LLM 判 — 更稳定、更可重复。
12

评估的核心价值

不是一刀切地判"合不合格"
而是发现问题和跟踪进步

找到 Agent 的薄弱维度, 每次迭代看分数有没有提升。
