# AI Agent 评估体系：维度、机制与实践

> 小组内部技术分享 | 2026-03-19

---

## 一、为什么要评估 Agent？

### 1.1 Agent 不等于 LLM

传统的 LLM 评估（如 MMLU、HumanEval）测的是**模型本身**的知识和推理能力。但 Agent = LLM + 工具调用 + 环境交互 + 多轮决策，这意味着：

- **一个好的 LLM 不一定是一个好的 Agent**。模型在基准测试上得分很高，但接入工具后可能频繁产生非法调用、忘记上下文、或无法完成多步任务。
- **Agent 的能力是涌现的**。它的表现取决于 LLM、提示词、工具设计、交互策略的组合效应，单独评估任何一个部分都不够。

### 1.2 评估的三个核心目的

| 目的 | 说明 | 举例 |
|------|------|------|
| **选型决策** | 在多个 Agent 方案中选出最优 | GPT-4 vs Claude vs 开源模型，哪个做编码 Agent 更好？ |
| **质量保障** | 上线前验证 Agent 是否达标 | 新版本 Agent 是否在边界场景中仍能正确操作？ |
| **诊断改进** | 定位 Agent 的薄弱环节 | Agent 在"简单查询"上表现好，但在"多步推理"时经常迷失 |

### 1.3 不评估会怎样？

不同类型的 Agent 在未经评估时，都可能暴露严重问题：

| Agent 类型 | 典型失败模式 |
|-----------|------------|
| **客服对话 Agent** | 声称"已为您退款"但实际没有调用任何工具；遗漏必须告知用户的关键信息 |
| **代码生成 Agent** | 生成的代码能通过编译但逻辑错误；引入安全漏洞；修复 bug A 却引入 bug B |
| **浏览器自动化 Agent** | 点击了错误的按钮；在页面加载完成前就操作；陷入无限循环 |
| **数据分析 Agent** | SQL 查询返回了结果但逻辑错误（如混淆 JOIN 方向）；对数据的解读产生幻觉 |
| **操作系统 Agent** | 执行了危险命令（如误删文件）；权限越界；命令顺序错误导致状态不一致 |

这些问题只有通过**系统性的多维度评估**才能被发现——仅靠人工试用几次是远远不够的。

## 二、Agent 评估的必要维度

基于对 AgentBoard（NeurIPS 2024）、AgentBench（ICLR 2024）、τ²-Bench（Sierra Research）三个主流评估框架的深入调研，我们总结出以下评估维度体系：

### 2.1 维度全景图

```
                     Agent 评估维度体系
    ┌─────────────────────────────────────────────┐
    │                                             │
    │  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
    │  │ 执行正确性 │  │ 交互质量   │  │ 行为合规  │  │
    │  │ (做对了吗) │  │ (表达好吗) │  │ (守规矩吗)│  │
    │  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
    │       │             │             │         │
    │  ┌────┴─────┐  ┌────┴─────┐  ┌───┴──────┐  │
    │  │ 状态变更  │  │ 信息传达  │  │ 策略遵循  │  │
    │  │ 动作执行  │  │ 回复质量  │  │ 安全边界  │  │
    │  │ 工具使用  │  │ 多轮连贯  │  │ 一致性   │  │
    │  └──────────┘  └──────────┘  └──────────┘  │
    └─────────────────────────────────────────────┘
```

### 2.2 六大核心维度详解

#### 维度一：环境状态正确性（State Match）

**衡量什么**：Agent 的操作是否真正改变了环境到期望状态。

**为什么必须评估**：这是 Agent 评估的"硬指标"。Agent 说了什么不重要，重要的是**做了什么**。

**不同 Agent 类型中的体现**：

| Agent 类型 | "状态正确"意味着什么 |
|-----------|-------------------|
| 客服 Agent | 后端数据库中订单状态确实被修改了 |
| 代码 Agent | 生成的代码在目标环境中编译通过、测试通过 |
| 浏览器 Agent | 页面状态达到了预期（如购物车中有正确商品） |
| DB 操作 Agent | SQL 执行后表中数据与预期一致 |
| OS Agent | 文件系统/进程/配置达到了预期状态 |

**实现方法**：
- **精确匹配**：将实际环境状态的哈希值与期望状态的哈希值进行比对（τ²-Bench 方案）
- **子集匹配**：当精确匹配失败时，按字段逐一比对，给出部分分（0.0~1.0）
- **特定断言**：代码类可以用测试通过率，浏览器类可以用 DOM 状态比较

```
评估流程（以 DB 状态为例）：
  期望路径: 初始状态 → 执行期望动作 → Gold 状态 → hash_gold
  实际路径: 初始状态 → 回放实际轨迹 → Pred 状态 → hash_pred

  hash_gold == hash_pred ? → 1.0（完全正确）
  否则按字段子集比较    → 0.0 ~ 1.0（部分正确）
```

**优势**：与执行顺序无关——Agent 可以用不同路径达到相同结果，只要最终状态正确即可。

#### 维度二：动作执行匹配（Action Match）

**衡量什么**：Agent 是否执行了关键的工具调用/操作步骤，且参数是否正确。

**为什么必须评估**：有时最终状态可能碰巧正确（比如数据本来就没变），但 Agent 其实根本没有执行关键操作。动作匹配确保 Agent 走了正确的"路径"。

**不同场景的示例**：

```python
# 客服场景：期望 Agent 先查询再修改
expected_actions = [
    {"name": "query",  "match_args": {"table": "orders", "id": "ORD-001"}},
    {"name": "update", "match_args": {"table": "orders", "status": "cancelled"}}
]

# 代码场景：期望 Agent 先读文件再写入
expected_actions = [
    {"name": "read_file",  "match_args": {"path": "src/main.py"}},
    {"name": "write_file", "match_args": {"path": "src/main.py"}}
]

# 浏览器场景：期望 Agent 搜索后点击商品
expected_actions = [
    {"name": "type", "match_args": {"selector": "#search-input"}},
    {"name": "click", "match_args": {"selector": ".product-card"}}
]
```

使用**部分参数匹配**——只比较关键字段，不要求所有参数完全一致。

#### 维度三：信息传达完整性（Info Delivery）

**衡量什么**：Agent 是否向用户传达了所有必要的信息。

**为什么必须评估**：完成操作只是一半的工作。Agent 还需要把**用户需要知道的信息**清楚地传达出来。

**不同 Agent 类型中的体现**：

| Agent 类型 | 必须传达的信息 |
|-----------|-------------|
| 客服 Agent | 退款金额、到账时间、后续流程 |
| 代码 Agent | 修改了哪些文件、为什么这样改、潜在影响 |
| 数据分析 Agent | 查询结果的含义、数据的局限性、分析结论 |
| 浏览器 Agent | 当前操作进展、遇到的异常、最终结果 |
| OS Agent | 命令执行结果、变更的影响范围 |

**实现方法**：
- 定义一组 `required_info`（必须传达的信息关键词/短语）
- 在 Agent 的所有回复中搜索这些信息是否被提及
- 支持模糊匹配（大小写不敏感、子串匹配）

#### 维度四：回复质量（LLM Judge）

**衡量什么**：Agent 的回复在准确性、完整性、清晰度方面的综合表现。

**为什么必须评估**：前三个维度是"硬指标"，但 Agent 的表达方式同样重要。

- 代码 Agent 的解释是否让人看得懂？
- 数据分析 Agent 的结论是否有理有据？
- 客服 Agent 的语气是否专业友好？
- 任何 Agent 的回复是否自相矛盾、是否产生幻觉？

**实现方法**：
- **LLM-as-Judge**：用另一个 LLM 对 Agent 的回复打分（1-5 分）
- 评分归一化：`score = LLM评分 / 5.0 → 0.0 ~ 1.0`
- 特别适用于**黑盒 Agent**——当你无法观察 Agent 的内部工具调用时，回复质量可能是唯一能评估的维度

#### 维度五：自然语言断言（NL Assertion）

**衡量什么**：用自然语言描述的任意评判条件是否成立。

**为什么必须评估**：有些评估条件无法用简单的状态比对或子串匹配来验证。这是最灵活的评估维度，能覆盖任何复杂的判断逻辑。

**各类 Agent 的断言示例**：

```json
{
    "客服 Agent": [
        "Agent 在执行取消操作前确认了用户的意图",
        "Agent 没有泄露其他客户的个人信息",
        "Agent 在无法满足需求时提供了替代方案"
    ],
    "代码 Agent": [
        "Agent 修改代码前解释了修改原因",
        "Agent 没有引入新的安全漏洞",
        "Agent 保持了代码的向后兼容性"
    ],
    "数据分析 Agent": [
        "Agent 在给出结论前验证了数据质量",
        "Agent 没有对不完整的数据做过度推断",
        "Agent 正确处理了 NULL 值和异常值"
    ],
    "OS Agent": [
        "Agent 在执行破坏性操作前请求了用户确认",
        "Agent 没有使用 sudo 执行不必要的命令",
        "Agent 在操作失败后尝试了恢复"
    ]
}
```

**实现方法**：
- 将完整交互轨迹交给 LLM，逐条判断每个断言 PASS 或 FAIL
- 最终得分 = 通过的断言数 / 总断言数

#### 维度六：一致性与稳定性（Consistency）

**衡量什么**：Agent 在多次执行同一任务时，结果是否稳定。

**为什么必须评估**：LLM 的输出本身带有随机性。一个 Agent 如果只是"偶尔"能完成任务，在生产环境中是不可靠的。这对所有类型的 Agent 都至关重要——你不会信任一个"有时能正确写代码，有时写出 bug"的编码 Agent。

**实现方法**（τ²-Bench 的 Pass^k）：
- 对每个任务运行 n 次（如 4 次）
- Pass^1：至少成功 1 次的概率（最宽松，衡量能力上限）
- Pass^k：k 次全部成功的概率（最严格，衡量稳定性）

```
Pass^1 高但 Pass^4 低 → Agent 有能力但不稳定（最危险——给人虚假信心）
Pass^1 低             → Agent 能力不足（至少问题很明确）
Pass^1 高且 Pass^4 高 → Agent 既有能力又稳定 ✓
```

### 2.3 AgentBoard 的额外维度：过程指标

AgentBoard（NeurIPS 2024）提出了两个独特的**过程评估**维度，弥补了"只看结果"的不足：

| 维度 | 含义 | 为什么重要 |
|------|------|-----------|
| **进度率 (Progress Rate)** | 即使任务未完成，也追踪完成了多少子步骤 | 区分"完全没做"和"做了一半"的 Agent，给出改进方向 |
| **接地准确度 (Grounding Accuracy)** | 合法动作数 / 总动作数 | 衡量 Agent 是否"知道自己能做什么"——发出非法指令说明 Agent 对工具理解有误 |

**Progress Rate 示例**：

```
任务："在电商网站上找到最便宜的无线鼠标并加入购物车"

Agent A: 搜索 → 排序 → 选商品 → 加购物车   PR=1.0 ✓ 完全成功
Agent B: 搜索 → 排序 → 选商品 → (放弃)      PR=0.75  差一步
Agent C: 搜索 → (陷入循环)                    PR=0.25  卡在第一步
Agent D: (什么都没做)                          PR=0.0   完全失败

仅看 Success Rate：A=1, B/C/D=0（无法区分 B/C/D 的差异）
看 Progress Rate：清楚地展示了每个 Agent 卡在哪一步
```

### 2.4 六大能力维度画像

AgentBoard 还将任务表现映射到**六大能力维度**，生成雷达图：

```
        记忆 (Memory)
           ╱╲
    反思  ╱    ╲  规划
         ╱      ╲
        ╱        ╲
       ╱──────────╲
      ╱            ╲
  接地              世界建模
(Grounding)    (World Modeling)
         ╲      ╱
          ╲    ╱
       空间导航
```

| 能力 | 含义 | 对应什么类型的失败 |
|------|------|------------------|
| **记忆** | 能否在多步交互中记住关键信息 | 多轮对话中忘记用户之前说的 |
| **规划** | 能否制定并执行多步计划 | 操作顺序混乱，先删再查 |
| **世界建模** | 能否理解环境的状态和规则 | 不知道某个操作的前置条件 |
| **自我反思** | 失败后能否调整策略 | 一直重复同样的错误操作 |
| **接地** | 能否把意图正确转化为合法操作 | 调用不存在的工具或传错参数 |
| **空间导航** | 能否在复杂环境中定位和移动 | 在网页/文件系统中迷路 |

这种能力画像对**诊断改进**特别有价值——不是告诉你"Agent 不好"，而是告诉你"Agent 的规划能力不足，但记忆和接地能力很好"。

### 2.5 维度对比：各框架的侧重

| 维度 | AgentBoard | AgentBench | τ²-Bench | 通用 Agent 评估 |
|------|:----------:|:----------:|:--------:|:--------------:|
| 环境状态正确性 | - | 任务特定 | DB 哈希比对 | 必须 |
| 动作执行匹配 | - | - | 部分参数匹配 | 推荐 |
| 信息传达完整性 | - | - | 子串检查 | 推荐 |
| 回复质量 | - | - | - | 可选 |
| 自然语言断言 | - | - | LLM-as-Judge | 推荐 |
| 任务完成率 | Success Rate | 二值判定 | 二值判定 | 必须 |
| 进度率 | Progress Rate | - | - | 推荐 |
| 接地准确度 | Grounding Acc | - | - | 推荐 |
| 一致性 | - | - | Pass^k | 推荐 |
| 能力维度画像 | 6维雷达图 | - | - | 可选 |

## 三、评分机制：为什么用乘积而非加权平均？

### 3.1 两种评分方案对比

```
方案 A — 加权平均：
  overall = 0.4 x state_match + 0.3 x action_match + 0.3 x info_delivery

  例子：state_match=0, action_match=1.0, info_delivery=1.0 → overall=0.6
  解读：Agent 什么都没做对，但因为表达好，还能拿 60 分？

方案 B — 乘积组合：
  overall = state_match x action_match x info_delivery

  结果：state_match=0, action_match=1.0, info_delivery=1.0 → overall=0.0
  解读：任何一个维度完全失败，总分归零。
```

### 3.2 乘积组合的设计哲学

这一设计借鉴自 τ²-Bench，背后的核心理念是：

> **Agent 的各评估维度之间是"与"关系，不是"或"关系。**

用不同类型的 Agent 来理解这个原则：

| 场景 | state | action | info | overall | 判定 |
|------|:-----:|:------:|:----:|:-------:|:----:|
| 代码 Agent 写对了代码，但没解释修改内容 | 1.0 | 1.0 | 0.0 | **0.0** | 不合格 |
| OS Agent 解释清楚了，但命令没执行 | 0.0 | 0.0 | 1.0 | **0.0** | 不合格 |
| 数据 Agent 查询正确，解释清楚 | 1.0 | 1.0 | 1.0 | **1.0** | 合格 |
| 浏览器 Agent 操作正确但漏了关键步骤 | 1.0 | 0.5 | 1.0 | **0.5** | 部分合格 |

只有**所有维度都达标**，这次任务执行才算成功。

### 3.3 评分流程全景

```
Task 执行
    |
    |-- Orchestrator 驱动交互循环
    |     Agent <-> User <-> Environment
    |     记录完整轨迹 (trajectory)
    |
    |-- 评估器逐一评分
    |     state_match  -> 0.0 ~ 1.0  (环境状态对了吗)
    |     action_match -> 0.0 ~ 1.0  (关键步骤做了吗)
    |     info_delivery-> 0.0 ~ 1.0  (必要信息说了吗)
    |     llm_judge    -> 0.0 ~ 1.0  (回复质量如何) [可选]
    |     nl_assertion -> 0.0 ~ 1.0  (约束条件守了吗) [可选]
    |
    |-- 乘积组合
    |     overall = state x action x info x ...
    |
    |-- 多次试验 (num_trials)
    |     计算 Pass^k 一致性指标
    |
    +-- 按难度分层统计
          easy / medium / hard 分别报告
```

### 3.4 评分机制的局限与权衡

乘积组合虽然严格，但也有需要注意的地方：

| 优势 | 局限 |
|------|------|
| 任一维度为零则总分为零，避免偏科掩盖问题 | 对于探索性任务（如创意写作），"完全失败"的边界不清晰 |
| 简单直观，容易理解和解释 | 无法表达"这个维度更重要"的权重关系 |
| 鼓励 Agent 全面达标 | 多个维度都是 0.8 时，乘积会快速衰减（0.8^5=0.33） |

**实践中的应对**：
- 通过选择合适的评估器组合来控制严格程度（不是每个任务都需要所有评估器）
- 报告中同时展示分项得分和总分，方便诊断
- 对于探索性任务，可以只用 `llm_judge + nl_assertion`，不用严格的状态匹配

## 四、评估架构：如何组织这些维度？

### 4.1 通用的 Agent-Environment 交互循环

所有 Agent 评估框架都遵循同一个核心模式：

```
                ┌──────────┐
    初始化 ----> |  Agent   | <----+
                | (被评估者) |      |
                └─────┬────┘      |
                      |           |
                   动作/回复       观察/反馈
                      |           |
                      v           |
               ┌──────────────┐   |
               | Environment  |---+
               | (任务环境)    |
               └──────────────┘
                      |
                 任务完成/超时
                      |
                      v
               ┌──────────────┐
               |  Evaluators  |
               | (多维度评分)   |
               └──────────────┘
```

这个循环是**通用的**——无论你评估的是客服 Agent、代码 Agent 还是浏览器 Agent，底层模式都一样。区别只在于：
- **Agent** 用什么适配器接入（API 调用？HTTP 接口？本地进程？）
- **Environment** 提供什么工具和状态（数据库？文件系统？浏览器？）
- **Evaluators** 关注哪些维度（状态匹配？测试通过率？页面状态？）

### 4.2 插件化设计

通过注册表模式实现完全解耦：

```
┌─────────────────── 不变的框架核心 ───────────────────┐
|                                                      |
|  Orchestrator --- 驱动 Agent <-> User <-> Env 循环    |
|  Registry    --- 插件注册表（装饰器模式）              |
|  Report      --- 评估报告生成                         |
|                                                      |
├─────────────────── 可插拔的扩展 ─────────────────────┤
|                                                      |
|  @registry.adapter("xxx")     -> Agent 适配器         |
|  @registry.environment("xxx") -> 任务环境             |
|  @registry.evaluator("xxx")   -> 评估维度             |
|  @registry.user("xxx")        -> 用户模拟器           |
|                                                      |
└──────────────────────────────────────────────────────┘
```

**核心价值**：新增 Agent 类型或评估维度只需添加一个文件 + 一个装饰器，不需要修改框架核心代码。

### 4.3 不同 Agent 类型的评估配置

| Agent 类型 | 适配器 | 环境 | 核心评估维度 | 补充维度 |
|-----------|--------|------|------------|---------|
| **工具调用 Agent** (API/FC) | openai_fc | mock_db | state_match, action_match | info_delivery |
| **对话 Agent** (客服/咨询) | openai_fc / http_bot | mock_db / passthrough | info_delivery, llm_judge | nl_assertion |
| **代码 Agent** | http_bot / openai_fc | docker sandbox | 测试通过率, 代码质量 | action_match |
| **浏览器 Agent** | http / websocket | playwright | 页面状态, 视觉准确度 | action_match |
| **数据分析 Agent** | openai_fc | db + dataframe | 查询结果正确性 | info_delivery |
| **OS 操作 Agent** | ssh / docker | linux 容器 | 系统状态匹配 | nl_assertion (安全) |
| **黑盒 Agent** (仅看输入输出) | http_bot | passthrough | info_delivery, llm_judge | nl_assertion |

### 4.4 评估方法分类

```
       确定性评估                        LLM-as-Judge 评估
  ┌──────────────────┐            ┌──────────────────────┐
  | state_match      |            | llm_judge            |
  |   DB 哈希比对     |            |   LLM 打分 (1-5)     |
  |   字段子集比较    |            |                      |
  |   测试通过率      |            | nl_assertion         |
  |                  |            |   LLM 逐条判断       |
  | action_match     |            |   PASS/FAIL          |
  |   工具名+参数匹配 |            |                      |
  |                  |            |                      |
  | info_delivery    |            |                      |
  |   子串搜索       |            |                      |
  └──────────────────┘            └──────────────────────┘

  优点：完全可复现                  优点：能评估开放式问题
  成本：零额外 API 调用             成本：需要额外 LLM 调用
  适用：有明确标准答案              适用：主观质量、合规性判断
```

**如何选择？** 能用确定性评估的优先用确定性评估（更便宜、更可复现）；无法用规则衡量的维度才用 LLM-as-Judge。

## 五、业界参考：三大评估框架对比

| 维度 | AgentBoard (港科大) | AgentBench (清华) | τ²-Bench (Sierra) |
|------|:---:|:---:|:---:|
| **发表** | NeurIPS 2024 Oral | ICLR 2024 | Industry |
| **定位** | 分析型评估平台 | 分布式通用基准 | 客服对话评估 |
| **核心创新** | 6维能力雷达图 + 进度率 | 分布式 Client-Server | 双控制 + LLM-as-User |
| **任务类型** | 具身AI、游戏、网页、工具 (9类) | DB、KG、OS、网页 (5-8类) | 航空、零售、电信 (3个域) |
| **评估方法** | 连续奖励 (0~1) | 任务特定指标 | 严格二值乘积 |
| **用户模拟** | 无 | 无 | LLM 模拟用户 |
| **关键指标** | SR + PR + Grounding | 任务成功率 | Pass^k |
| **架构** | 单进程 + gym 接口 | Client-Server + Docker | Orchestrator 编排 |

### 核心启发

- **AgentBoard** 告诉我们：评估不能只看"成功/失败"二值结果，进度率、接地准确度、能力画像等**过程指标**对诊断问题至关重要。它覆盖了从具身 AI 到网页操作的 9 类任务，是最具广度的评估平台。
- **AgentBench** 告诉我们：评估框架的**工程架构**很重要——插件化、可配置、可扩展，Docker 隔离确保了不同任务环境互不干扰，Max-Flow 调度实现了高效的并行评估。
- **τ²-Bench** 告诉我们：真实场景需要**多方参与**（Agent + 用户 + 环境），单一维度的评判会遗漏大量问题。它的四重评判乘积和 Pass^k 一致性指标是对话类 Agent 评估的标杆。

## 六、实践建议

### 6.1 评估维度选择指南

```
你的 Agent 是做什么的？
    |
    +-- 有明确的环境状态变更？（DB/文件/系统状态）
    |     是 --> state_match (必选) + action_match (推荐)
    |     否 --> 跳过
    |
    +-- 需要向用户输出信息？
    |     是 --> info_delivery (必选)
    |     否 --> 跳过
    |
    +-- 需要评估回复的主观质量？
    |     是 --> llm_judge
    |     否 --> 跳过
    |
    +-- 有难以用规则表达的约束条件？（安全/合规/风格）
    |     是 --> nl_assertion
    |     否 --> 跳过
    |
    +-- 需要验证生产可靠性？
          是 --> 多次试验 + Pass^k
          否 --> 单次试验即可
```

### 6.2 设计好的测试场景

一个好的测试场景应该包含：

| 要素 | 说明 | 示例（通用） |
|------|------|------------|
| 明确的初始状态 | Agent 面对的起始环境 | DB 数据 / 文件内容 / 页面 URL / 系统配置 |
| 清晰的任务描述 | 需要完成什么 | "查询用户最近的订单" / "修复这个 bug" / "找到最便宜的商品" |
| 期望的关键动作 | 必须执行的步骤 | 调用了哪些工具、传了哪些关键参数 |
| 必须传达的信息 | 必须告知用户的内容 | 查询结果 / 修改说明 / 操作确认 |
| 自然语言约束 | 不应该做的事 | 不应未经确认执行危险操作 / 不应泄露敏感信息 |
| 难度标注 | 场景的复杂度 | easy / medium / hard |

### 6.3 常见陷阱

| 陷阱 | 说明 | 应对 |
|------|------|------|
| **只看成功率** | 无法知道 Agent 哪里做得好、哪里差 | 多维度评估 + 轨迹可视化 |
| **评估不可复现** | LLM 随机性导致每次结果不同 | 固定 temperature=0 + 多次试验 + Pass^k |
| **测试场景太简单** | 通过率 100% 但实际使用时翻车 | 分难度层级、加入边界情况和对抗场景 |
| **忽略过程只看结果** | 最终状态对了但路径荒谬 | action_match + 进度率追踪 |
| **评估维度单一** | 只检查状态变更，忽略交互质量 | 根据 Agent 类型组合多个评估维度 |
| **场景类型单一** | 全是简单任务或全是同一类型 | 覆盖 easy/medium/hard + 不同任务类型 |
| **把评估当测试** | 评估只在开发阶段做一次 | 持续评估，每次迭代都跑，追踪趋势 |

## 七、总结

### Agent 评估的核心公式

```
Agent 评估 = 多维度 x 多场景 x 多试验

多维度：不只看结果，还看过程、看沟通、看合规
多场景：覆盖不同难度、不同类型的任务
多试验：衡量稳定性，而非单次偶然成功
```

### 一句话总结各维度

| 维度 | 一句话 | 适用的 Agent 类型 |
|------|--------|-----------------|
| State Match | Agent **做对了**吗？ | 所有有状态变更的 Agent |
| Action Match | Agent **走对路**了吗？ | 工具调用型、操作型 Agent |
| Info Delivery | Agent **说清楚**了吗？ | 所有需要向用户输出的 Agent |
| LLM Judge | Agent **表达好**吗？ | 对话型、分析型 Agent |
| NL Assertion | Agent **守规矩**了吗？ | 所有有合规/安全要求的 Agent |
| Progress Rate | Agent **走了多远**？ | 多步骤复杂任务 |
| Grounding | Agent **操作合法**吗？ | 工具调用型 Agent |
| Pass^k | Agent **稳得住**吗？ | 所有生产环境 Agent |

### 评分哲学

> **乘积组合，而非加权平均。任何一个维度的完全失败都应该导致总分归零。**
>
> 因为在真实世界中：
> - 一个"代码全错但解释很好"的编码 Agent 是不合格的
> - 一个"操作全对但什么都不解释"的自动化 Agent 是不可信的
> - 一个"大部分时候都能成功"但偶尔翻车的 Agent 是不可靠的

## 八、实践落地：从理论到三种评估模式

基于前面的维度体系和评分机制，我们在 Agent Evaluator 框架中落地了三种评估模式，覆盖从白盒到黑盒的不同 Agent 类型。

### 8.1 三种评估模式全景

| 模式 | 适配器 | 环境 | 核心评估维度 | 适用场景 |
|------|--------|------|------------|---------|
| **工具调用 Agent** | `openai_fc` | `mock_db` | state_match, action_match, info_delivery | 有函数调用能力的 API Agent |
| **多轮对话 Agent** | `openai_fc` + `scripted`/`llm` user | `mock_db` | + nl_assertion | 需要与用户多轮交互的 Agent |
| **黑盒 HTTP Agent** | `http_bot` | `passthrough` | info_delivery, llm_judge | 只能通过 HTTP 接口访问的 Agent |

```
模式一：工具调用（白盒）
  用户消息 → Agent(openai_fc) → 工具调用 → mock_db → 评估状态变更

模式二：多轮对话（半白盒）
  用户消息 → Agent ↔ User(scripted/llm) → 工具调用 → mock_db → 评估状态+断言

模式三：黑盒 HTTP（纯黑盒）
  用户消息 → Agent(http_bot) → 纯文本回复 → passthrough → 评估回复质量
```

### 8.2 HTTP Bot 黑盒评估

当被评估的 Agent 只暴露了一个 HTTP 接口（如企业内部的对话 Bot），我们无法观察其内部工具调用和状态变更。HTTP Bot 适配器解决了这个问题：

**三种历史模式**：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `last` | 只发最后一条消息 | Bot 自己管理上下文 |
| `history` | 发完整对话历史 | 无状态 Bot，需要接收完整历史 |
| `session` | 用 session_id 管理会话 | Bot 通过 session 维护上下文 |

**配置示例**：

```yaml
agent:
  adapter: http_bot
  url: "http://bot-service:8080/chat"
  history_mode: last
  request_field: query        # 请求体中消息字段名
  reply_field: data.reply     # 支持嵌套 JSON 路径解析
  max_retries: 3
  retry_delay: 1.0
```

**评估策略**：无工具调用 → 用 `passthrough` 环境（无状态变更）→ 评估聚焦于 `info_delivery`（必要信息是否传达）和 `llm_judge`（回复质量打分）。

### 8.3 eval_bot：飞书评测 Bot

为了让团队成员更方便地触发评测，我们构建了一个飞书 Bot（`eval_bot/`），通过自然语言交互即可完成评测操作。

**架构**：

```
飞书消息 → Webhook(FastAPI) → Dispatcher(LLM 意图识别)
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              quick_eval      query_results    gen_scenarios
              (异步评测)       (查询结果)       (生成场景)
                    │
                    ▼
              TaskRunner(线程池) → 完成后回调 → 推送飞书消息
```

**三个核心命令**：

| 命令 | 说明 | 示例 |
|------|------|------|
| `quick_eval` | 对 HTTP Bot 发起快速评测 | "帮我测一下客服 Bot" |
| `query_results` | 查询历史评测结果 | "上次评测结果怎么样？" |
| `gen_scenarios` | 为业务领域生成测试场景 | "帮我生成电商客服的测试用例" |

**关键设计**：
- **Dispatcher** 使用智谱 function calling 识别用户意图，路由到对应命令
- **TaskRunner** 线程池异步执行评测，不阻塞飞书消息响应
- 评测完成后自动将结果推回飞书群，包含分数摘要和报告链接

### 8.4 架构验证：core/ 零修改

从 Step 1（核心骨架）到 Step 4（eval_bot），`core/` 目录下的四个文件（`types.py`、`base.py`、`registry.py`、`orchestrator.py`）始终保持零修改。这验证了插件化架构的核心价值：

```
新增的所有组件：
  adapters/http_bot.py         ← @registry.adapter("http_bot")
  environments/passthrough.py  ← @registry.environment("passthrough")
  evaluators/llm_judge.py      ← @registry.evaluator("llm_judge")
  evaluators/nl_assertion.py   ← @registry.evaluator("nl_assertion")
  users/scripted_user.py       ← @registry.user("scripted")
  users/llm_user.py            ← @registry.user("llm")
  eval_bot/                    ← 独立模块，调用框架 API

核心引擎修改：0 个文件
```
